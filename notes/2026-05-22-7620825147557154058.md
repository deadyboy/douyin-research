## 一句话概括

当 AI 技能数量庞大时，别依赖大模型“猜”，而是用 CSV 做知识库、BM25 做检索、规则引擎做决策——构建一个轻量、可解释、可复用的智能体系统。

---

## 这个视频在讲什么

这是一段屏幕录制的软件教程视频，展示了一个名为“UI UX Pro Max”的智能体系统架构。它不依赖大模型或向量数据库，而是通过结构化的 CSV 文件、经典的 BM25 检索算法和明确的规则引擎，来处理多达 37 个技能的复杂调用。视频重点讲解了如何用工程化手段管理大量技能，实现从用户输入到设计系统输出的完整闭环。

---

## 关键内容拆解

视频的核心是一个“无模型、纯工程”的智能体架构。它把技能管理拆解为三个关键部分：

1. **知识库用 CSV**：所有设计规则、配色方案、UI 风格、技术栈推荐等，都存成 13 个 CSV 文件（如 `styles.csv`, `colors.csv`, `charts.csv`），总计 1453 行数据。这种结构清晰、易读、易维护，适合轻量级系统。

2. **检索用 BM25**：用户输入关键词（如“beauty spa wellness service”），系统用 BM25 算法在 CSV 中做文本匹配，找出最相关的条目。相比向量搜索，它不需要训练、无上下文限制、完全离线运行。

3. **决策用规则引擎**：匹配结果会进入 `ui-reasoning.csv`（含 161 条规则），按“优先级分类”（如可访问性 > 性能 > 布局）进行聚合和推理，最终生成结构化的设计系统文档。

整个流程是：用户输入 → BM25 检索 → 规则匹配 → 聚合输出 → 生成设计系统。所有步骤都可调试、可复现，避免了大模型的“幻觉”和上下文溢出。

## 为什么值得关注

这个方案打破了“智能体必须依赖大模型”的惯性思维。它用最基础的工具（CSV + Python + BM25）实现了复杂任务，特别适合以下场景：

- 技能/规则数量多、结构清晰、更新频率低（如 UI 设计规范、技术选型指南）。
- 对可解释性、可维护性要求高（如团队协作、审计、调试）。
- 需要离线运行、无网络依赖、无 GPU 资源（如部署在本地或边缘设备）。

更重要的是，它提供了一种“工程化智能体”的范式：把 AI 任务拆解为“数据 + 算法 + 规则”的组合，而不是依赖黑箱模型。

## 可以怎么复用

你可以直接照搬这套架构，快速搭建自己的轻量智能体系统：

1. **创建目录结构**：
   ```bash
   mkdir skills && mkdir skills/data && touch skills/SKILL.md skills/data/styles.csv
   ```

2. **定义技能文档**：在 `SKILL.md` 中写明每个技能的“触发条件”、“工作流”、“优先级分类”和“快速参考”。

3. **构建 CSV 知识库**：把你的规则、配置、推荐项整理成 CSV 文件，比如 `colors.csv`、`fonts.csv`、`tech-stack.csv`。

4. **实现 BM25 检索器**：
   ```python
   from sklearn.feature_extraction.text import TfidfVectorizer
   import pandas as pd
   df = pd.read_csv('data/styles.csv')
   vectorizer = TfidfVectorizer()
   tfidf_matrix = vectorizer.fit_transform(df['Description'])
   query = 'minimalist'
   query_vec = vectorizer.transform([query])
   from sklearn.metrics.pairwise import cosine_similarity
   scores = cosine_similarity(query_vec, tfidf_matrix).flatten()
   top_indices = scores.argsort()[-5:][::-1]
   print(df.iloc[top_indices])
   ```

5. **封装命令行工具**：写一个 `search.py`，支持参数输入，输出推荐结果。

6. **设计聚合模块**：写 `design_system.py`，按优先级分类聚合结果，生成最终输出。

7. **添加快速参考**：在 `SKILL.md` 末尾加一个“快速参考”章节，列出常用命令和输出格式。

这套方法不仅适用于 UI 设计，也适用于技术选型、文档生成、配置推荐等任何结构化规则密集的场景。

## 需要注意的边界

虽然这套方案轻量高效，但也有适用边界：

- **不适合高度语义化任务**：如果用户输入模糊、跨领域混合（如“做一个既像苹果又像谷歌的电商网站”），BM25 和规则引擎可能无法准确理解意图。

- **规则复杂度有限**：`ui-reasoning.csv` 中的 161 条规则未展示具体内容，无法判断其覆盖范围。如果规则逻辑过于复杂，维护成本会升高。

- **未展示实际输出**：视频未展示生成的设计系统文档或前端代码，无法验证最终效果。实际使用中需自行测试输出质量。

- **BPOS 算法需查证**：视频中“BPOS”与“BM25”交替出现，但未明确定义。可能是笔误或自定义实现，需进一步确认。

总之，它是一个优秀的“工程化智能体”范例，适合规则明确、结构清晰的场景，但不适合需要深度语义理解的任务。
